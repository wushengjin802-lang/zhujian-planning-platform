# 总体架构

## 架构分层

```text
前端应用
  工作台、项目、资料、事实、报告、质量、成果、知识、系统

API 服务
  认证权限、项目、文件、解析、事实、报告、质量、成果、审计

异步任务
  文档解析、OCR、知识索引、章节生成、质量检查、成果导出

数据层
  业务数据库、对象存储、向量索引、审计日志

模型与外部服务
  LLM 网关、Office 导出、GIS 服务、Excel 计算服务
```

## 第一阶段实现策略

当前正式主线已按 PDD 技术栈回改：

- `apps/web-vue` 负责界面、导航和 MVP 主链路视图，使用 Vue3、Element Plus、Pinia、Vue Router；
- `apps/api-py` 提供统一 bootstrap 数据和关键资源 API，使用 FastAPI、SQLAlchemy、Pydantic；
- `apps/api-py/alembic` 管理 PostgreSQL `zhujian/public` 的数据库迁移；
- `apps/api-py/app/worker` 提供 Celery 任务入口，按 parse、ai、export、calc 队列分流；
- `infra/docker/docker-compose.pdd.yml` 提供 Redis 与 MinIO 本地开发依赖；
- 原 React + Express 实现已移至 `archive/react-express-prototype`，仅作为页面信息架构和历史接口参考，不再作为正式第一阶段主线扩展。
- `/api/platform/status` 暴露数据库扩展、Redis、MinIO、LibreOffice、模型网关状态，用于验收 PDD 技术栈落地情况。

## PostgreSQL 接入

第一阶段核心业务库使用 PostgreSQL。当前数据库为 `zhujian`，schema 为 `public`，已覆盖：

- `projects`：项目基本信息；
- `project_documents`：资料清单、版本和解析状态；
- `fact_items`：事实与指标、来源、责任人和确认状态；
- `report_chapters`：报告章节、引用数和审核状态；
- `quality_issues`：质量问题和发布门禁；
- `artifacts`：成果文件状态；
- `parse_jobs`、`quality_check_jobs`：异步任务占位；
- `audit_logs`：操作审计。

## 文件与成果存储

第一阶段对象存储主线为 MinIO/S3。开发环境未启动 MinIO 时，后端会降级到本地 `storage/`，以便不阻塞资料上传与成果导出验证：

- `storage/documents`：上传的项目资料；
- `storage/artifacts`：生成的成果占位文件。

资料上传后记录 `storage_path`、`file_size`、`mime_type` 和 `checksum`。成果导出会先检查未关闭阻断问题，通过后生成可下载文件。

## 模型与 RAG

章节生成走“模型网关优先、本地 fallback 兜底”的路径：

- 已配置 `MODEL_GATEWAY_URL` 时，后端向统一模型网关提交 prompt 与检索上下文；
- 网关未配置或不可用时，使用本地规则生成可审计初稿，避免阻塞第一阶段链路；
- 资料切片通过轻量检索参与上下文构建，pgvector 安装后可进一步升级为向量召回与重排序。

## Office 与 PDF 转换

成果生成使用 `python-docx`、`openpyxl`、`python-pptx` 生成 Word、Excel、PPT 和归档包。安装 LibreOffice 或配置 `LIBREOFFICE_PATH` 后，归档包会附带 PDF 转换结果。

## 扩展状态

Alembic 迁移会尝试启用 `PostGIS` 和 `pgvector`。当前远端数据库服务端缺少 PostGIS 扩展包时会跳过扩展创建，业务表迁移仍会完成。GIS 空间分析和向量检索需要 DBA 安装并授权扩展后再补建空间字段、embedding 字段和索引。
